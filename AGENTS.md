# tech-notes

このリポジトリは技術ノートのまとめ場所です。issue が 1 トピックのノートに対応し、`@Codex` メンション付きの issue に対して技術的な内容を調査・整理してコメントします。

## 回答のルール

- 必ず日本語で回答すること
- issue のタイトル・本文で問われている技術トピックについて、正確で実用的な解説をコメントとして残すこと
- 不確かな情報は不確かであると明記すること

## コメントのフォーマット

以下の構成でコメントを書くこと:

1. **概要** — トピックの要点を 2〜3 文で。図で説明できるトピック(処理の流れ、システム構成、状態遷移、クラス関係、シーケンスなど)の場合は、この概要のなかで mermaid.js の図を添えて視覚的に説明すること
2. **何が嬉しいのか** — この技術が解決する課題と、使わなかった場合と比べて何が良くなるのか。可能なら具体的なユースケースを挙げる
3. **詳細** — 仕組み・使い方・注意点など。コード例が有用な場合は含める
4. **参考リンク** — 公式ドキュメントを優先

### 図(mermaid.js)について

- GitHub は issue コメント内の mermaid をそのまま描画するため、` ```mermaid ` のコードブロックで記述すること
- 図が理解の助けにならないトピック(用語の定義だけ、など)では無理に図を入れないこと
- 図の種類は内容に合わせて選ぶ(`flowchart` / `sequenceDiagram` / `stateDiagram-v2` / `classDiagram` / `erDiagram` など)
- ラベルに日本語を使う場合、記号によっては構文エラーになることがあるため、必要に応じてラベルを `"..."` で囲むこと

## タイトル編集のルール

issue 作成時のタイトルは暫定的なものが入っていることが多い。最初のコメントを残す際に、`gh issue edit <番号> --title "<タイトル>"` でノートの内容を表す適切なタイトルに変更すること。

- タイトルは日本語で、トピックが一目で分かる簡潔なものにする(例: 「React useEffect のクリーンアップ処理」)
- すでに内容を適切に表すタイトルが付いている場合は変更しなくてよい
- 2 回目以降のコメント(追加質問への返信)ではタイトルは変更しない。ただし議論の結果ノートの主題が大きく変わった場合は更新してよい

## ラベル付与のルール

コメントを残したら、`gh issue edit <番号> --add-label "<ラベル>"` で issue にラベルを付与すること。

- ラベルは既存のものから選ぶこと。`gh label list` で確認できる。**新しいラベルを勝手に作らないこと**
- ジャンルラベル(`frontend`, `backend` など)を 1 つ以上付与すること
- 種類ラベル(`type:` プレフィックス)をちょうど 1 つ付与すること

### ジャンルラベルの選び方

| ラベル | 対象 |
|---|---|
| `frontend` | React、CSS、ブラウザ、UI 関連 |
| `backend` | サーバーサイド、API 設計 |
| `infra` | クラウド、コンテナ、ネットワーク |
| `database` | RDB、NoSQL、SQL チューニング |
| `language` | プログラミング言語の仕様・文法そのものの話 |
| `ai-ml` | LLM、機械学習 |
| `security` | 認証認可、脆弱性、暗号 |
| `devops` | CI/CD、監視、開発ツール |
| `architecture` | 設計パターン、システム設計 |
| `cs-fundamentals` | アルゴリズム、データ構造、OS、低レイヤ |

### 種類ラベルの選び方

- `type:concept` — 概念・仕組みの解説
- `type:howto` — 手順・Tips
- `type:troubleshooting` — エラー・問題解決
- `type:comparison` — 技術比較・選定
