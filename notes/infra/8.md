# Dockerfile の基本文法まとめ

## 概要

Dockerfile は、Docker イメージをビルドするための手順をテキストで記述する設定ファイルです。1 行 1 命令の形式で「ベースイメージの指定」「ファイルのコピー」「コマンドの実行」などを上から順に定義し、`docker build` がこれを解釈してイメージを生成します。

## 何が嬉しいのか

Dockerfile を使わずに手動で `docker run` → コンテナ内で環境構築 → `docker commit` という手順を踏むと、構築手順が属人化・ブラックボックス化し、再現性がなくなります。Dockerfile があることで、

- 環境構築手順が **コードとしてバージョン管理** できる(Git で差分管理・レビュー可能)
- `docker build` 一発で **誰でも同じ環境を再現** できる
- CI/CD パイプラインに組み込んで **自動でイメージをビルド** できる

といったメリットが得られます。例えば「本番環境だけ Node のバージョンが違う」といった環境差異の事故を防げます。

## 詳細

### 代表的な命令

| 命令 | 役割 |
|---|---|
| `FROM` | ベースイメージを指定(Dockerfile の必須の最初の命令) |
| `RUN` | イメージビルド時にコマンドを実行し、その結果を新しいレイヤーとして確定 |
| `COPY` | ホスト上のファイル/ディレクトリをイメージにコピー |
| `ADD` | `COPY` に加え、tar の自動展開・URL からの取得も可能(基本は `COPY` 推奨) |
| `WORKDIR` | 以降の命令の作業ディレクトリを設定(なければ作成される) |
| `ENV` | 環境変数を設定(コンテナ実行時にも引き継がれる) |
| `ARG` | ビルド時のみ有効な変数(`docker build --build-arg` で上書き可能) |
| `EXPOSE` | コンテナがリッスンするポートをドキュメント目的で宣言(実際の公開は `docker run -p`) |
| `CMD` | コンテナ起動時のデフォルトコマンド(`docker run` の引数で上書き可能) |
| `ENTRYPOINT` | コンテナ起動時に必ず実行されるコマンド(上書きされにくい) |
| `USER` | 以降の命令やコンテナ実行時のユーザーを指定 |
| `VOLUME` | 永続化・共有したいディレクトリを宣言 |
| `LABEL` | メタデータ(作者、バージョンなど)をイメージに付与 |

### 基本的な例

```dockerfile
# ベースイメージを指定
FROM node:20-slim

# 作業ディレクトリを設定
WORKDIR /app

# 依存関係の定義だけ先にコピーしてキャッシュを効かせる
COPY package.json package-lock.json ./
RUN npm ci

# アプリ本体をコピー
COPY . .

# コンテナが待ち受けるポートを宣言(ドキュメント目的)
EXPOSE 3000

# コンテナ起動時に実行するコマンド
CMD ["node", "server.js"]
```

### 注意点

- **レイヤーキャッシュ**: 各命令ごとにレイヤーが作られ、変更がない命令はキャッシュが再利用されます。上記例のように「変更頻度が低いもの(依存関係のインストール)」を先に、「変更頻度が高いもの(アプリコード)」を後に書くとビルドが速くなります。
- **CMD と ENTRYPOINT の違い**: `CMD` は `docker run` に引数を渡すと丸ごと上書きされますが、`ENTRYPOINT` は固定のコマンドとして扱われ、`CMD` はその引数(デフォルト値)として使われる、という組み合わせ方もよく使われます。

  ```dockerfile
  ENTRYPOINT ["node"]
  CMD ["server.js"]
  ```

  この場合 `docker run image` は `node server.js` を実行し、`docker run image app.js` なら `node app.js` に変わります。
- **shell 形式と exec 形式**: `RUN`, `CMD`, `ENTRYPOINT` はどちらの記法も可能です。
  - shell 形式: `RUN npm install`(内部で `/bin/sh -c` 経由で実行される)
  - exec 形式: `RUN ["npm", "install"]`(シェルを介さず直接実行、シグナルの扱いが素直なため `CMD`/`ENTRYPOINT` では exec 形式が推奨されることが多い)
- **`.dockerignore`**: `.gitignore` と同様の書式で、`COPY . .` などでイメージに含めたくないファイル(`node_modules`, `.git` など)を除外できます。ビルドコンテキストが軽くなりビルドも速くなります。
- **マルチステージビルド**: `FROM ... AS builder` のように名前付きステージを複数作り、最終的に必要な成果物だけを次のステージに `COPY --from=builder` することで、ビルドツールなどを含まない軽量な本番イメージを作れます(基本文法の範囲を超える内容ですが、実務では非常によく使われるので知っておくと良いです)。
- **`ADD` より `COPY` を優先**: `ADD` は tar 展開や URL 取得といった暗黙的な挙動があり意図しない動作の原因になりやすいため、単純なファイルコピーには `COPY` を使うのが推奨されています。
