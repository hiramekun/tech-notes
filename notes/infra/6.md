# Docker マルチステージビルドの基本

## 概要

マルチステージビルド(multi-stage build)は、1 つの Dockerfile 内に複数の `FROM` を書き、ビルド用ステージと実行用ステージを分離する仕組みです。ビルドに必要なコンパイラや依存パッケージを最終的なイメージに含めず、実行に必要な成果物だけを次のステージへコピーすることで、軽量で安全な本番イメージを作れます。Docker 17.05 以降で利用可能です。

## 何が嬉しいのか

マルチステージビルドを使わない場合、次のような問題が起きます。

- ビルドツール(コンパイラ、`gcc`、`node_modules` のビルド依存、`.git` など)が最終イメージに残り、**イメージサイズが肥大化**する
- ビルド時にしか使わないツールが本番環境に残ることで、**攻撃対象領域(attack surface)が増える**
- ビルド専用イメージと実行用イメージを別々の Dockerfile で管理する、あるいは中間生成物を手動でコピーするなど、**運用が煩雑になる**

マルチステージビルドを使うと、1 つの Dockerfile で完結しつつ、最終イメージには実行に必要なファイルだけが残ります。

**具体的なユースケース例:**
- Go や Rust のようなコンパイル言語で、ビルド環境(SDK 一式)は大きいが、実行に必要なのは単一バイナリだけの場合
- Node.js / TypeScript で `npm install` の devDependencies やビルドツール(webpack, tsc など)を使ってビルドし、実行時は成果物(`dist/`)と本番依存だけを含めたい場合
- テストやリンティングを専用ステージで実行し、最終イメージに影響を与えないようにしたい場合

## 詳細

### 基本構文

```dockerfile
# --- ビルドステージ ---
FROM golang:1.22 AS builder
WORKDIR /app
COPY . .
RUN go build -o myapp .

# --- 実行ステージ ---
FROM alpine:3.19
COPY --from=builder /app/myapp /usr/local/bin/myapp
ENTRYPOINT ["myapp"]
```

- `AS builder` でステージに名前を付ける
- `COPY --from=builder` で、前のステージのファイルシステムから必要なファイルだけを取り出す
- 最終ステージのベースイメージ(ここでは `alpine`)には Go のツールチェーンは一切含まれない

### Node.js の例

```dockerfile
FROM node:20 AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-slim AS runtime
WORKDIR /app
COPY --from=build /app/dist ./dist
COPY --from=build /app/node_modules ./node_modules
COPY package*.json ./
CMD ["node", "dist/index.js"]
```

### 応用テクニック

- **外部イメージからのコピー**: `COPY --from=nginx:latest /etc/nginx/nginx.conf .` のように、自分のビルドステージ以外のイメージからもファイルを取得できる
- **特定ステージだけビルド**: `docker build --target builder -t myapp:debug .` のように `--target` オプションで途中のステージだけをビルドし、デバッグ用イメージとして使うことも可能
- **キャッシュの活用**: 依存関係のインストール(`COPY package*.json ./` → `RUN npm ci`)とソースコードのコピーを分けることで、ソース変更時に依存関係の再インストールを避けキャッシュを効かせられる

### 注意点

- 最終ステージのベースイメージ選定も重要(`alpine` や `distroless` など、実行に必要な最小限のものを選ぶとさらに軽量・安全になる)
- `COPY --from` はステージ番号(`--from=0`)でも指定できるが、可読性のため名前付きステージ(`AS builder`)を使うことを推奨
- ビルドステージが増えると Dockerfile がやや複雑になるため、CI での `docker build` 時間とのバランスも考慮する
